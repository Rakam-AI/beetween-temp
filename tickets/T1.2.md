# T1.2 - Remove Greeting Sample Domain


**Depends on:** T1.1
**Blocks:** T1.7

**Context:**
The template contains a sample "Greeting" domain used to demonstrate the hexagonal architecture pattern. This includes domain models (`Greeting`, `GreetingId`), use cases (`CreateGreetingUseCase`, `CreateGreetingApplicationService`), ports (`GreetingRepositoryPort`, `TimeProviderPort`, `MetricsPublisherPort`, `OutboxPort`), adapters (`GreetingController`, `NoOpGreetingRepositoryAdapter`, etc.), DTOs, domain events, and tests.

**Objective:**
Remove all Greeting-specific code while preserving the architectural skeleton, error handling framework, configuration, and security setup.

**Instructions:**
1. **Delete the following files completely:**
   - `domain/model/Greeting.kt`, `domain/model/GreetingId.kt`
   - `domain/errors/DomainError.kt` (will be recreated with project-specific errors)
   - `domain/events/DomainEvent.kt`, `domain/events/GreetingDone.kt`
   - `usecase/commands/CreateGreetingCommand.kt`
   - `usecase/service/CreateGreetingApplicationService.kt`
   - `usecase/service/shared/SomeInternalApplicationService.kt`
   - `usecase/ports/inbound/CreateGreetingUseCase.kt`
   - `usecase/ports/outbound/GreetingRepositoryPort.kt`
   - `usecase/ports/outbound/TimeProviderPort.kt`
   - `usecase/ports/outbound/MetricsPublisherPort.kt`
   - `usecase/ports/outbound/OutboxPort.kt`
   - `usecase/model/GreetingResponse.kt`
   - `usecase/errors/UseCaseError.kt` (will be recreated)
   - `adapters/inbound/web/GreetingController.kt`
   - `adapters/inbound/web/dto/GreetingResponseDto.kt`
   - `adapters/outbound/persistence/NoOpGreetingRepositoryAdapter.kt`
   - `adapters/outbound/messaging/NoOpOutboxAdapter.kt`
   - `adapters/outbound/monitoring/MicrometerMetricsPublisherAdapter.kt`
   - `adapters/outbound/systemtime/SystemTimeProviderAdapter.kt`

2. **Keep intact:**
   - `Application.kt` (main entry point)
   - `config/BeanConfig.kt` (remove Greeting bean wiring, keep the class)
   - `config/security/SecurityWebConfig.kt`
   - `config/security/ClientLoggingInterceptor.kt`
   - `config/security/actuate/KeycloakHealthIndicator.kt`
   - `config/openapi/OpenApiConfig.kt` (update title/description for DAR)
   - `config/utility/AppConfig.kt`, `config/utility/Helper.kt`
   - `adapters/inbound/web/errormanager/` (all error handling files)
   - `application.yml`, `docker-compose.yml`, `Dockerfile`, `pom.xml`

3. **In `config/BeanConfig.kt`:** Remove all `@Bean` methods that reference Greeting classes. Leave the class with `@Configuration` annotation as an empty shell.

4. **In `config/openapi/OpenApiConfig.kt`:** Update the API title to "DAR API - Demand Approval Request" and description accordingly.

5. **Update test files:**
   - Delete `domain/GreetingTest.kt`
   - Delete `it/system/greeting/GreetingIT.kt`
   - Keep `architecture/CleanArchitectureTest.kt` but update package references from Greeting to the new root
   - Keep `architecture/NamingRulesTest.kt`, `ApiSurfaceRulesTest.kt`, `ApplicationServiceRulesTest.kt` (update as needed)
   - Keep `IntegrationTestBase.kt` and `IntegrationTestConfig.kt`

**Acceptance Criteria:**
- Project compiles with `./mvnw clean compile` (even if some ArchUnit tests fail due to missing classes, that's expected)
- No references to "Greeting", "greeting", or "hello" remain in `src/main/`
- Error handling framework still intact
- Security configuration still intact
- OpenAPI config updated
