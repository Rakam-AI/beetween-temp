# T2.9 - Implement Request Rejection Use Case


**Depends on:** T2.4, T2.6
**Blocks:** T2.13

**Context:**
Any validator at the current step can reject the request, regardless of validation mode. Rejection is final (no correction possible, must duplicate).

Reference:
- `API_Technical Specification V1.md` > POST /requests/{id}/rejection
- `Agent_Technical Specification V1.0.md` > "Etape 3b: Rejet par le/les responsables"

**Objective:**
Create the RejectRequestUseCase.

**Instructions:**

1. **Create `RejectRequestUseCase` interface and `RejectRequestService` implementation:**
   - Load request (validate exists, PENDING status)
   - Load workflow, validate user is validator at current step
   - Reason is mandatory (return error if blank)
   - Call `request.reject(rejectorId, rejectorType, reason, comment)`
   - Save request
   - Publish `RequestRejected` event

2. **Write unit tests:**
   - Reject with reason → REJECTED
   - Reject without reason → error
   - Reject by non-validator → error
   - Reject when not PENDING → error

**Acceptance Criteria:**
- Rejection requires a reason
- Status transitions to REJECTED
- currentStepIndex does NOT change
- History entry includes reason and comments
- Event published with all required fields
